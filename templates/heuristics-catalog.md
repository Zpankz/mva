## Heuristic-001 — [Short, Memorable Title]

**Cycles:** 003, 007, 012

**Statement (Imperative or Declarative):**
"The workflow orchestration engine must enforce saga-level idempotency on start commands, not merely task-level idempotency. Duplicate saga initiation requests must resolve to the same saga instance."

**Evidence (What We Observed Under Stress):**
"In cycle 007, a retried `StartSaga` command (due to a transient network blip between the API gateway and the orchestrator) created two parallel saga instances for the same business intent. Both proceeded to the payment step, resulting in double-billing. The downstream payment service was itself idempotent, but the saga orchestrator had no concept of 'this business request already has a saga in flight.'"

**Implication for Governance / Rebuild:**
- Saga start operations must be deduplicated using a business-keyed idempotency store that lives at the orchestrator level.
- The idempotency key must be derived from the initiating business command, not generated by the orchestrator.
- Any implementation that treats saga start as fire-and-forget without upstream idempotency will re-create this class of failure.

**Applies To:**
- Orchestration layer
- Saga runtime
- API gateway / command ingestion layer (in how it generates/plumbs idempotency keys)

**Confidence:** High

**Status:** Incorporated into governance-layer.md §1.2 and §5
